## 1. Incident → Event → Log

### Definition

**Incident**
- A security-relevant occurrence that impacts or has the potential to impact an organization
- Can be:
  - **Intentional** — attacks, insider threats
  - **Unintentional** — misconfiguration, human error

**Event**
- Any observable activity in a system or network
- Includes normal operations and security-related actions

**Log**
- A record of events generated by systems, applications, or devices
- Stored in a structured format for monitoring, auditing, and investigation

### Explanation (Core Concept)

This is a **hierarchy**:

> Event → Log → Incident (after analysis)

- Systems generate **events**
- Events are recorded as **logs**
- Analysts analyze logs to identify **incidents**

### Examples

| Type | Example |
|------|--------|
| **Incident** | Brute force attack against login portal |
| **Event** | User login success, User login failure |
| **Log** | Failed login attempt from 192.168.1.10, Successful login from 192.168.1.10 |

### SOC Relevance

- SOC analysts **DO NOT** directly see incidents
- They see:
  - **Logs**
  - **Alerts**
  - **Events**
- Their job is to:
  1. **Analyze logs**
  2. **Correlate events**
  3. **Determine if it is an incident**

### Detection Perspective

**Key Pattern**: Multiple failed logins → success → possible brute force

**Logs Used**:
- Authentication logs
- Windows Event Logs (4625, 4624)
- VPN logs

**Indicators of Compromise (IoCs)**:
- High number of failed logins
- Same IP across multiple accounts
- Login success after failures
- Unusual login timing (night hours)

### Investigation Approach (SOC L1)

1. Identify source IP
2. Check login pattern
3. Verify user behavior
4. Check geo-location
5. Escalate if suspicious

### Interview Points

**Q: Difference between Event and Incident?**
- **Event** = any activity
- **Incident** = confirmed security threat

**Q: Do all events become incidents?**
- No, only suspicious or malicious ones

## 2. Typical Log Sources

### Definition

Log sources are systems/devices that generate logs for monitoring and analysis.

### Common Log Sources

| Category | Log Sources |
|----------|-------------|
| **Endpoint Logs** | Windows Event Logs, Linux Syslogs, Antivirus logs |
| **Network Logs** | Firewall logs, IDS/IPS logs, VPN logs |
| **Application Logs** | Web server logs (Apache, IIS), Database logs, Email logs |
| **Security Devices** | SIEM logs, EDR logs, NAC logs |

### Explanation

Every component in IT infrastructure generates logs:
- **Endpoints** → user activity
- **Network devices** → traffic flow
- **Applications** → user interaction
- **Security tools** → threats
- **SOC** combines all into central SIEM

### SOC Relevance

- Without logs → **no visibility**
- More log sources → **better detection capability**

### Detection Perspective

| Source | Detects |
|--------|--------|
| Firewall | Blocked IPs |
| Windows Logs | Login activity |
| Web Logs | SQL injection |
| DNS Logs | Domain beaconing |
| EDR | Malware execution |

**IoCs**: Suspicious IPs, Unknown domains, Abnormal traffic spikes, Unauthorized access attempts

### Interview Points

**Q: Most important log source?**
- Depends, but **authentication + endpoint + firewall logs** are critical

## 3. Need for Logging

### Definition

Logging is required to monitor, detect, investigate, and comply with security requirements.

### Key Needs

| # | Need | Description |

| 6 | **Compliance** | Required for: GDPR, HIPAA, PCI-DSS |

### SOC Relevance

- **Logging = foundation of SOC**
- No logs → **SOC is blind**

### Detection Perspective

- Compare: Normal vs abnormal behavior
- **Example**: User logs in daily at 9 AM → suddenly logs in at 3 AM

### Interview Points

**Q: Why are baselines important?**
- To detect anomalies

## 4. Logging Requirements

### Before Enabling Logging

You must define:
1. **What to log** — Authentication, network traffic, admin actions
2. **Where to store logs** — SIEM / log server
3. **Logging methods** — Syslog, agents, APIs
4. **Tools** — Splunk, QRadar, ELK
5. **Log format** — JSON, CEF, etc.

### Capabilities Required

1. **Regular Review** — Continuous monitoring
2. **Synchronization** — Time sync (NTP) — Critical for correlation
3. **Log Integrity Protection** — Prevent tampering
4. **Correlation** — Combine multiple log sources
5. **Analysis** — Detect threats

### SOC Relevance

- Bad logging = missed attacks
- Poor sync = incorrect timelines

### Detection Perspective

- **Example**: Firewall log + endpoint log + DNS log = full attack chain
- **IoCs**: Missing logs, Log gaps, Timestamp mismatch

### Interview Points

**Q: Why is time synchronization important?**
- To correlate events accurately across systems

## 6. Logging Approaches

### 1. Local Logging

**Definition**: Logs stored on the same system where events occur.

**Examples**:
- System start/stop
- Account creation/deletion
- Disk space issues
- Application updates

**Problems**:
- Can be deleted by attacker
- Limited visibility

### 2. Centralized Logging

**Definition**: Logs sent to a central server (SIEM).

**Examples**:
- Firewall logs
- VPN logs
- Admin activity logs
- Network access logs

**Advantages**:
- Single view of environment
- Easier correlation
- More secure (harder to tamper)

### SOC Relevance

- SOC always uses **centralized logging**
- SIEM = heart of SOC

### Detection Perspective

- Combine logs: Endpoint + Network + Identity
- **IoCs**: Log forwarding stopped, Missing logs from device, Sudden silence from host

### Investigation Approach

1. Check log source status
2. Verify log ingestion in SIEM
3. Investigate missing logs
4. Look for tampering

## 17. Monitoring Linux Logs using Auditd

### Definition

**Auditd** (Linux Audit Daemon) is a host-based logging system used to:
- Track security-relevant events
- Record user activities
- Provide forensic-level logging

### Core Components

| Component | Description |
|-----------|-------------|
| **auditd** | Background daemon that collects and writes logs |
| **auditctl** | Tool to define rules |
| **ausearch** | Search audit logs |
| **aureport** | Generate summaries |

### Log Location

- `/var/log/audit/audit.log`

### Explanation

Unlike normal logs (syslog), **Auditd provides deep visibility**:
- Tracks: File access, System calls, User actions, Privilege escalation
- Used for: Compliance (PCI-DSS, HIPAA), Forensics

### Example Log Entry (Simplified)

```
type=SYSCALL msg=audit(1620000000): user=root exe="/usr/bin/passwd"
```

### What Auditd Can Monitor

- File changes (`/etc/passwd`)
- User login/logout
- Command execution
- Permission changes
- Network access

### SOC Relevance

- Detect: Insider threats, Privilege escalation, File tampering

### Detection Perspective

**Example: Privilege Escalation**
- User runs: `sudo su`
- Audit log: Command execution, UID change
- **IoCs**:
  - Access to sensitive files: `/etc/shadow`
  - Unauthorized sudo usage
  - Suspicious command execution

### Investigation Approach

1. Search logs: `ausearch -ua 1001`
2. Generate report: `aureport -au`
3. Check: User activity, Command history, File access

### Interview Points

**Q: Difference between syslog and auditd?**

| Syslog | Auditd |
|--------|-------|
| General logging | Security-focused |
| Less detailed | Deep visibility |
| No syscall tracking | Tracks syscalls |

## 26. Firewall Logs

### Definition

Firewall logs record network traffic activity passing through a firewall, including:
- Allowed traffic
- Blocked traffic
- Connection attempts

### Explanation

- Firewalls sit between: Internal network (trusted) ↔ External network (untrusted)
- Every packet is: **Allowed**, **Denied**, or **Dropped** — All these actions are logged

### Why Firewall Logs Matter

- Provide visibility into network traffic
- Help detect: Attacks, Intrusions, Unauthorized access

### SOC Relevance

Firewall logs are one of the most critical data sources for SOC because they show:
- Who is connecting
- From where
- To which system
- Whether it was allowed or blocked

### Detection Perspective

**Key Questions Analysts Ask**:
- **Who** (Source IP)?
- **Where** (Destination IP)?
- **What** (Port/Protocol)?
- **Action** (Allow/Deny)?

**Key IoCs**: Repeated blocked connections, Port scanning activity, Connections to suspicious IPs, Traffic to unusual ports

### Interview Points

**Q: Why are firewall logs important?**
- They provide network-level visibility and attack detection

---

## 27. Monitoring & Analysis of Firewall Logs

### Steps of Analysis

| Step | Action |
|------|--------|
| **Step 1** | **Locate Logs** — Local system or SIEM |
| **Step 2** | **Identify Key Fields** — Source IP, Destination IP, Port, Protocol, Action |
| **Step 3** | **Interpret Traffic** — Incoming vs outgoing, Normal vs suspicious |
| **Step 4** | **Analyze Action** — Allowed or blocked |
| **Step 5** | **Trace Source IP** — Use WHOIS, GeoIP tools |

### SOC Relevance

- Daily SOC task: Investigating alerts based on firewall logs

### Detection Example

- Multiple denied connections → port scan
- Allowed connection to malicious IP → compromise

**Internet Information Services (IIS)** is a web server used on Windows systems to:
- Host websites
- Serve web applications
- Record user activity via logs

### Log Location (Important)

| IIS Version | Location |
|-------------|---------|
| **IIS 6.0** | `%SystemRoot%\System32\LogFiles\W3SVCN` |
| **IIS 7+** | `%SystemDrive%\inetpub\logs\LogFiles` |

### What IIS Logs Capture

- User IP address
- Date & time
- Requested URL
- HTTP method (GET, POST)
- Response status
- User agent (browser info)

### Explanation

Each request to a website is logged:
```
Client → Web Server → Response → Log entry
```

### SOC Relevance

- Detect: Web attacks, Unauthorized access, Bot activity, Data exfiltration

### Monitoring & Analysis (IMPORTANT)

| Field | Description |
|-------|-------------|
| **c-ip** | Client IP |
| **date & time** | Timestamp |
| **cs-method** | GET/POST |
| **cs-uri-stem** | Requested resource |
| **sc-status** | Response code |
| **cs(User-Agent)** | Browser |

### Detection Examples

1. **SQL Injection**: `GET /login.php?id=1' OR '1'='1`
2. **Directory Traversal**: `GET /../../etc/passwd`

### IoCs

- Suspicious URLs
- Repeated requests
- Unusual user agents
- High 404/500 errors

### Investigation Approach

1. Identify suspicious IP
2. Check request pattern
3. Analyze status codes
4. Correlate with firewall logs

### Interview Points

**Q: What does sc-status 404 mean?**
- Resource not found

## 46. Why Centralized Logging?

### Definition

**Centralized logging** is the practice of collecting logs from multiple systems into a single central location for monitoring and analysis.

### Problem Without Centralization

| # | Issue | Description |
|---|-------|-------------|
| 1 | **Every system generates logs independently** | Scattered data |
| 2 | **Each device can generate thousands of logs per day** | Volume problem |
| 3 | **Analysts must manually check multiple systems** | Not scalable |

### Explanation

Without centralization:
- Logs are scattered
- Hard to correlate events
- High chance of missing attacks

### SOC Relevance

- SOC cannot function efficiently without centralized logging
- It is the foundation of **SIEM**

### Real-World Example

- Attack spans: Firewall → Endpoint → Server
- Without central logging → impossible to connect events

### Interview Points

**Q: Why centralized logging is needed?**
- To reduce complexity and enable correlation

## 60. Step 5: Log Correlation

### Definition

**Log correlation** is the process of:
- Linking multiple normalized log events
- Identifying relationships between them
- Detecting patterns that indicate an attack

### Explanation

| Signal Type | Description |
|-------------|-------------|
| **Single log** | Weak signal |
| **Correlated logs** | Strong evidence |

### Example

- Firewall → Allowed connection
- Web server → Suspicious request
- Endpoint → Malware execution
- **Individually**: normal
- **Together**: incident

### Types of Correlation

| Type | Description | Example |
|------|-------------|--------|
| **Micro-Level (Atomic)** | Works on single event or small event set, Based on field matching, Rule-based detection | Multiple failed logins from same IP |
| **Macro-Level** | Combines: Threat intelligence, Geo-location, Blacklists, Vulnerability data | Login from malicious IP (TI feed), Suspicious country |

### Correlation Methods

- **Rule-based** (most common in SOC)
- Statistical correlation
- Algorithmic / ML-based correlation

### SOC Relevance

- Core function of Splunk, IBM QRadar
- Converts noise → meaningful alerts

### Detection Perspective

**Example: Brute Force Attack**
- Event 1: 20 failed logins
- Event 2: 1 success login
- **Correlation rule**: IF `failed_logins > 10` AND `success_login` THEN alert

### IoCs

- Repeated patterns across logs
- Same IP across multiple systems
- Sequential attack behavior

### Investigation Approach

1. Identify correlated events
2. Build timeline
3. Identify root cause
4. Confirm attack chain

### Interview Points

**Q: Why correlation is important?**
- Detects attacks that single logs cannot reveal
